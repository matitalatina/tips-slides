How to build a reusable UI

with Angular

Starting point

the mockup from our beloved designer

1. Atoms

Build smallest UI components

  • State only for UI purposes
  • Inputs must be very generic: label, title, hidden, disabled, item
  • Outputs must be very generic: clicked, closed, focused
  • No models from backend should be used inside
  • Goal: maximize reusability

Examples atomic UI components

  • Button
  • Input
  • ListItem

2. Molecules

Build UI components using atoms

  • State only for UI purposes
  • The data are given from inputs (no API calls yet)
  • Wire atomic components together to build more complex UI
  • Inputs and Outputs are still very generic
  • Goal: encapsulate UI logic + maximize reusability

Examples of molecules

  • SearchBox
  • List
  • TabsView
  • Menu

3. Organisms

Build UI components using molecules

  • State only for UI purposes
  • No API calls yet
  • You start seeing backend models (user, bookings...) in Inputs and Outputs
  • Wire atoms and molecules to build UI for your business
  • Goal: encapsulate complex business logic

Examples of organisms

  • UserEditForm
  • BookingList
  • EventCalendar

4. Containers

Add life to organisms!

  • No UI here
  • Wrap organisms to add smart components
  • Heavy usage of dependency injection
  • Link your preferred state manager to UI
  • API calls happens here
  • Goal: add business logic to UI

Examples of container

  • <YOUR_ORGANISM>Container

5. Layouts

The skeleton of your page

  • @ViewChild, @ViewChildren and <ng-content> are your best friends here
  • Goal: reuse common layouts, add consistency to your pages

Examples of layouts

  • TwoColumnsLayout
  • RootLayout (w/ Header and Footer)

6. Pages

Link components to Angular Route

  • The main wrapper is usually a layout
  • Wrap organisms to add smart components
  • Angular router builds pages
  • Goal: wire angular routes to UI

Why humans prefer
functional programming

  • It focuses on what you want to achieve
  • Fewer lines of code
  • It encourages code reusability (composing map, flatMap, filter, reduce etc.)
  • It hides implementation details

Why machines prefer
imperative programming

  • It focuses on how you want to achieve it
  • You write what the machine should do
  • Better performance (not always)

When you write code
remember:

  • Someone will review it
  • Someone should be able to understand it
  • Someone will take inspiration from that
  • Someone will build on top of it

Hardware is cheap,
developers are expensive

  • It's better to save hours spent by developers instead of milliseconds spent by machines
  • You should help your team building things quickly

So write human-friendly code

  • Functional programming can help you
  • Your code will be easier to understand and easier to change
  • Your colleagues will appreciate!
